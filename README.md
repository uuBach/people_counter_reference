# Entry Camera People Counter — референсный baseline

Это простое референсное решение для ML/CV intern task. Цель решения — быть запускаемым, понятным для проверки и честным по ограничениям, а не идеально точным production-решением.

## Файлы проекта

```text
run.py
requirements.txt
README.md
results/summary.json       # создается после запуска
results/events.csv         # создается после запуска
results/debug_frames/      # опционально, создается с --save-debug
results/tracks_map.jpg     # опционально, создается с --save-debug
```

Ожидаемый путь к видео:

```text
data/entry_camera.mp4
```

Видео не добавляется в репозиторий, так как оно предоставлено отдельно в рамках тестового задания и не должно распространяться.

```bash

```

## Подход

### Детектор

Скрипт использует публичную pretrained YOLO-модель из Ultralytics. По умолчанию загружается:

```text
yolov8n.pt
```

Используется только класс `person` из COCO. Модель `yolov8n.pt` выбрана потому, что она маленькая, быстро запускается и подходит для простого baseline-решения.

Для лучшей точности можно использовать модель крупнее, например:

```bash
python run.py --model yolov8s.pt --imgsz 1280
```

Компромисс: более крупная модель и больший размер изображения обычно лучше находят маленьких или частично перекрытых людей, но работают медленнее.

### Трекер

Для трекинга используется ByteTrack через Ultralytics:

```text
bytetrack.yaml
```

Трекер присваивает каждому обнаруженному человеку `track_id` и пытается сохранять этот ID между кадрами. Это нужно, чтобы не считать одного и того же человека заново на каждом кадре.

### Логика определения направления

По условию задания:

```text
upper-left/checkpoint side -> lower-right/site side = entry
lower-right/site side -> upper-left/checkpoint side = exit
```

То есть движение из верхней левой части кадра в нижнюю правую считается входом, а движение в обратном направлении — выходом.

Для каждого завершенного трека скрипт делает следующее:

1. Берет несколько первых center points трека и усредняет их.
2. Берет несколько последних center points трека и усредняет их.
3. Считает нормализованное смещение:

```text
dx = (end_x - start_x) / frame_width
dy = (end_y - start_y) / frame_height
```

4. Так как координаты изображения растут вправо и вниз, движение из upper-left в lower-right означает, что `dx + dy > 0`.
5. Если смещение достаточно сильное и движение похоже на диагональное, трек классифицируется как:

```text
entry: dx + dy > 0
exit:  dx + dy < 0
```

Если движение слишком слабое, короткое или неоднозначное, трек получает класс:

```text
unknown
```

### Unknown / ignored tracks

Трек помечается как `unknown`, если он выглядит ненадежным для подсчета. Основные случаи:

* слишком мало детекций;
* трек слишком короткий по времени;
* человек почти не двигался;
* движение не похоже на направление entry/exit;
* слишком маленькое диагональное смещение;
* возможный дубликат после ID switch.

За это отвечают параметры командной строки:

```bash
--min-hits 5
--min-track-seconds 1.0
--min-movement-norm 0.06
--min-diagonal-delta 0.05
--min-alignment 0.45
```

### Обработка дублей

Иногда трекер может разделить одного человека на несколько разных `track_id`. Это называется ID switch.

В скрипте есть консервативная постобработка. Если второй принятый трек:

* имеет то же направление;
* начинается вскоре после окончания первого;
* начинается рядом с точкой, где закончился первый трек;

то второй трек помечается как `unknown`, потому что он может быть фрагментом того же самого человека.

Эта логика не идеальная. Она специально сделана осторожной, чтобы случайно не объединить двух разных людей, которые идут близко друг к другу.

## Frame sampling

По умолчанию используется:

```bash
--frame-sample-rate 5
```

Видео имеет 20 FPS, значит при таком параметре детекция и трекинг идут примерно на 4 FPS.

Зачем нужен frame sampling:

* видео длится 30 минут;
* всего в видео 36,000 кадров;
* запуск YOLO на каждом кадре может быть медленным, особенно на CPU;
* обработка каждого 5-го кадра — практичный baseline.

Компромисс:

* высокий sample rate, например 10, работает быстрее, но может терять короткие треки и чаще вызывать ID switch;
* низкий sample rate, особенно 1, работает медленнее, но обычно дает лучшее качество трекинга.

Для финального прогона, если хватает времени, лучше использовать:

```bash
--frame-sample-rate 1
```

## Результаты

После запуска скрипт сохраняет:

```text
results/summary.json
results/events.csv
```

Пример `summary.json`:

```json
{
  "video": "entry_camera.mp4",
  "entries": 0,
  "exits": 0,
  "unknown_tracks": 0,
  "frame_sample_rate": 5,
  "processed_frames": 7200,
  "total_video_frames": 36000,
  "notes": "YOLO person detector + ByteTrack..."
}
```

`events.csv` содержит одну строку на каждый классифицированный трек:

```text
track_id,direction,start_time_s,end_time_s,confidence,notes
```

Возможные значения `direction`:

```text
entry
exit
unknown
```

## Debug artifacts

При запуске с параметром:

```bash
--save-debug
```

создаются дополнительные файлы:

```text
results/debug_frames/*.jpg
results/tracks_map.jpg
```

Annotated frames показывают найденных людей и их track IDs.
`tracks_map.jpg` показывает траектории треков и стрелки направления.

Эти файлы нужны для проверки, насколько правдоподобно работает подсчет, без ручного hard-code ответа.

## Основные failure cases

Основные случаи, где решение может ошибаться:

1. **Перекрытия людей у checkpoint booth**
   Если люди перекрывают друг друга, трекер может потерять человека или поменять ID.

2. **Короткие частичные треки**
   Если человек появляется только у края кадра, направление может быть неоднозначным.

3. **Статичные рабочие или охрана**
   Люди, которые стоят рядом с booth, могут детектироваться, но их не нужно считать как entry/exit. Для этого используются thresholds по движению.

4. **Группы людей**
   Если несколько людей идут близко друг к другу, detector/tracker может слить их в один трек или разделить одного человека на несколько треков.

5. **Упрощенная диагональная логика**
   Решение предполагает, что описание из задания верное: entry — это движение из upper-left в lower-right. Если реальная геометрия камеры немного другая, лучше добавить зоны или настроить thresholds.

6. **Frame sampling может пропускать быстрые события**
   Sampling ускоряет обработку, но может ухудшать tracking.

## Что можно улучшить при наличии большего времени

* Добавить явные entry и exit zones вместо одной диагональной логики.
* Настроить параметры ByteTrack именно под эту камеру.
* Использовать более сильную YOLO-модель или detector, дообученный на construction-site данных.
* Сделать небольшой validation set с вручную просмотренными событиями.
